The Method Component uses Equivalence Class Partitioning, a powerful and logical way of defining test case data.

If you are new to Vizitest, we recommend watching the whole video to see how to test a Method. You can also use the buttons below to jump to specific points in the video or docs.

Method Component - Docs

Equivalence Class Partitioning

A technique for dividing input and output data into sets of equivalent values. This is useful for generating test cases. This will make a lot more sense if you watch the video.

Equivalence Classes

See how to work with Equivalence Classes. An Equivalence class is a set of data, whose Representative Values are all considered to yield the same result when used for a method input parameter. When used in the Output section of the component, it contains a set of values, all of which are considered valid.

Representative Values

See how to work with Representative Values, which belong to Equivalence Classes. Often, a single value within an Equivalence Class is sufficient for Test Case creation. Values will be available in the Test Case Table.

Input - method parameters

The INPUT column contain a list of all method parameters. Each parameter will usually have a valid and an invalid Equivalence Class.

Return Values

The Return Values deals with the data returned by the method. There is usually a valid Equivalence Class that contains all possible return values and an invalid Equivalence Class.

Exceptions

If the method throws an Exception, you can choose from a list of all Exceptions or use the general Exception for any exception.

Test Case Table

Once you have configured your Inputs and Outputs (and assuming you're not using Custom Assertions or Side Effects), you might want to look at the Test Case Table. This should already be on the canvas.

Custom Assertions

You can write your own Custom Assertion code to evaluate the data returned and assert a true or false result.

Side Effects

Used in combination with the Instance Component, this allows you to detect whether the method modified an Instance. A Test Case in the Test Case Table can then examine the side effect to determine if the test should pass or fail.
